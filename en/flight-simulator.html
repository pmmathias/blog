Chapter 2

Phase 1 — The Scaffold

A Blank Canvas

Every project starts with a blank canvas. In our case: a terminal, npm create vite@latest, and the decision to go with Three.js as the 3D engine. No Unity, no Unreal — pure JavaScript code that runs directly in the browser.

Why Three.js? Because it has the lowest barrier to entry. No plugin, no app store, no download. The user opens a URL and flies. That was the vision from the start.

The first scene was simple: a camera, a sky, light.

const scene = new THREE.Scene();
const sky = new Sky();
sky.scale.setScalar(450000);
scene.add(sky);

const camera = new THREE.PerspectiveCamera(60, w/h, 2, 8000);
const renderer = new THREE.WebGLRenderer({ antialias: true });

In 20 minutes, the first empty scene with a blue sky was up. No terrain, no trees, no bird. But a beginning. And a beginning is all you need to build momentum.

Chapter 3

Phase 2 — A World from Nothing

How do you build a landscape without placing a single polygon by hand?

The answer: with math. More precisely, with parabolas. Imagine a flat table. Now place invisible hills on top — each defined by a center point, a radius and a height. At every point on the terrain you ask: how many of these hills overlap here? And add up their contributions.

function getTerrainHeight(x, z, arcs) {
  let h = 0;
  for (const arc of arcs) {
    const dx = x - arc.cx, dz = z - arc.cz;
    const distSq = dx * dx + dz * dz;
    const contribution = 1 - distSq / (arc.radius * arc.radius);
    if (contribution > 0) h += arc.height * contribution;
  }
  return h;
}

900 positive arcs form the hills. 270 negative arcs carve out valleys. A radial falloff ensures the island gently slopes into the sea at the edges. The result: a natural-looking landscape that looks slightly different every time you load it — but always plausible.

But a beautiful landscape is useless if it stutters. Computing 900 arcs per vertex? With a 400×400 grid, that’s over 140 million operations — per frame. That won’t fly.

Octree and Frustum Culling

The solution: an Octree divides the world into spatial cells. Instead of rendering all 100,000 trees and 900 buildings every frame, the engine simply asks: which cells are actually in the camera’s field of view? Frustum Culling discards everything outside the view cone. Out of 100,000 trees, typically only 3,000–5,000 are actually drawn.

Summary

900 parabolas + negative arcs + radial island falloff = a natural landscape, without ever placing a single vertex by hand.

Chapter 4

Phase 3 — The Bird Takes Flight

A Camera with a Fear of Heights

The terrain was there. But a bird that doesn’t fly is just a camera with a fear of heights.

The first approach was a simple arcade model: press a key, go up. Let go, sink down. It worked — technically. But it felt like an elevator with a panoramic window. No gliding, no lift, no flying.

So we built an aerodynamic model. The key insight: lift depends on the square of the velocity. That’s dynamic pressure — the same physics that keeps real birds and aircraft in the air.

const dynamicPressure = 0.5 * AIR_DENSITY * speed * speed;
const liftMag = dynamicPressure * wingArea * CL / BIRD_MASS;
velocity.y += baselineLift * dt;  // Wing incidence angle
velocity.y += GRAVITY * dt;       // -9.81 m/s²

The first wing flap produced… nothing. Gravity won. Every time. The bird fell like a rock, no matter how furiously you hammered the spacebar.

The problem: we had forgotten the wing incidence angle. Real bird wings aren’t flat — they have a built-in angle of attack of about 3°. This angle generates minimal lift even in level flight. Without it, the wing is a plank, and a plank falls.

It took six iterations. Six versions of the physics model until the balance of gravity, lift and wing flapping was right. Until you could press the spacebar and feel the bird climbing. Until a dive felt dangerous and a loop made your stomach clench.

Summary

Realistic physics ≠ fun physics. We needed 6 iterations before flapping actually gained altitude.

Chapter 5

Phase 4 — Hands Become Wings

The Wild Part

And then came the wild part. The idea: what if you didn’t use the keyboard, but your arms? What if you flapped your arms in front of the webcam and the bird in the game actually climbed?

MediaPipe Pose Detection makes this possible. Google’s ML model detects 33 body landmarks in real time — right in the browser, no server, no cloud. Shoulders, elbows, wrists. Everything we need.

The logic: we measure the height of both wrists relative to the shoulders. High hands = wings up. Low hands = wings down. The rate of change between them = wing flap.

const delta = Math.abs(recentAvg - olderAvg);
if (delta > 0.025) {
  flapStrength = clamp(delta * 8, 0.5, 1);
}

The challenges were subtler than expected. The webcam image is mirrored — left is right. Sometimes landmarks disappear because the arm is outside the frame. And detection accuracy fluctuates with lighting.

Our solution: Graceful Degradation. When your hands leave the frame, the bird glides on gently — instead of crashing. When MediaPipe detects no pose, the system silently falls back to keyboard controls. No error, no popup, no frustration.

The feeling when you raise your arms for the first time and the bird climbs — that’s the moment the project tips from “technically interesting” to “I want to show this to people.”

Chapter 6

The Last 20%

80% of the Time

The last 20% takes 80% of the time. That’s universal. And in this project, the last 20% was mostly about one thing: textures.

Five Layers of Terrain

A single-color landscape looks like Play-Doh. So we built a custom shader with five texture layers: sand on the beaches, grass on the hills, earth at mid-elevation, rock on the steep slopes, snow on the peaks. Each layer blends smoothly into the next.

float grassFactor = smoothstep(sandEnd, sandEnd + 8.0, h)
                  * (1.0 - smoothstep(grassEnd - 8.0, grassEnd + 8.0, h));
float rockFactor = smoothstep(earthEnd, earthEnd + 5.0, h)
                  * (1.0 - smoothstep(rockEnd - 8.0, rockEnd + 8.0, h));
float snowFactor = smoothstep(rockEnd - 8.0, rockEnd + 8.0, h);

The snow texture was gray. We spent three hours hunting the bug in the shader logic. We checked the blending factors, the height thresholds, the UV coordinates. Everything looked correct. Until we realized: the logic was correct. The texture just wasn’t white enough. A 200×200 pixel JPEG with an average value of 180 instead of 250. Three hours of debugging for an image editing problem.

Buildings and Settlements

An island without civilization feels empty. So we generated buildings — five types: houses, barns, towers, hotels and high-rises. Each with procedural brick or concrete textures, random proportions and color variations.

The placement uses noise functions: settlements emerge in flat areas near the coast. Forests take over the hills and mountains. A second noise layer separates urban zones from forest zones, so trees don’t grow through rooftops.

Summary

Snow-covered peaks, 900 buildings, 100,000 trees — all procedurally generated.

Chapter 7

How Real Ocean Waves Come to Life — on the GPU

Why flat water looks flat

The first water prototype in the simulator was a flat plane with an animated normal map — the standard recipe. The surface seems to move, but fly low and scan the horizon: the illusion gives out. The sea is geometrically flat. No crests, no troughs, no silhouette. A trembling sheet of concrete.

Real oceans look different. They have volume. Waves cast shadows on each other, sunlight refracts off every single wave, and the horizon is not a clean line but a jagged band. If you want to simulate that in real time, you have to rebuild the physics.

The idea: waves as a sum of frequencies

The core insight comes from oceanography: real sea waves are not a single wave but the superposition of thousands of frequencies. A light breeze creates thousands of tiny capillary waves, a storm generates large slow swells, and all of them superimpose to form what we perceive as „the sea“. It’s exactly the same phenomenon as in audio: an orchestra is not a single tone but the sum of hundreds of frequencies.

British mathematician Owen Phillips formulated a statistical spectrum in 1957 that describes how much wave energy to expect at which wavelength and direction — depending on wind speed, wind heading and fetch (the distance over which the wind has blown):

$$ P(\mathbf{k}) = A \, \frac{\exp\!\bigl(-1/(kL)^2\bigr)}{k^4} \, \bigl|\hat{\mathbf{k}} \cdot \hat{\mathbf{w}}\bigr|^2 $$

Here \(\mathbf{k}\) is the wave vector in frequency space, \(L = v^2/g\) the characteristic wavelength at wind speed \(v\), and \(\hat{\mathbf{w}}\) the wind direction. Crucially: this statistic lives in frequency space. Each point in a 2D grid corresponds to a wave frequency and direction — not a position in the water.

From formula to Pixar ocean

The first attempt to render ocean waves in computer graphics came in 1986 from Alain Fournier and William Reeves (Pixar) with „A Simple Model of Ocean Waves“ at SIGGRAPH — Gerstner-wave-based, not FFT. A year later, in 1987, Gary Mastin, Peter Watterberg and John Mareda published „Fourier Synthesis of Ocean Scenes“ in IEEE Computer Graphics and Applications — the first FFT-based ocean simulation in computer graphics. They showed how to turn a statistical wave spectrum (Pierson-Moskowitz back then) into a real height map via a Fourier transform.

The breakthrough for film and games came only in 2001, however, when Jerry Tessendorf of ILM published his SIGGRAPH course notes „Simulating Ocean Water“ — to this day the standard reference on the subject. Tessendorf didn’t invent the method, but he summarised it so cleanly and adapted it so directly to the needs of film production that his paper became the de-facto textbook.

Since then the method has lived in practically every AAA ocean: Titanic (even before Tessendorf’s paper), Pirates of the Caribbean, Life of Pi, Moana, Sea of Thieves, Assassin’s Creed Black Flag — all use FFT waves built on the same mathematical foundation.

Why the FFT is the decisive trick

At a resolution of 512×512 you need to superimpose 65,536 waves into a single coherent surface. Every frame. Naive individual addition would be \(O(N^2)\) and far too slow — \(2.7 \cdot 10^9\) operations per frame. The FFT does it in \(O(N \log N)\) — about \(1.2 \cdot 10^7\) operations. A factor of 1,000 faster.

The algorithm works in stages. For 512 samples that’s \(\log_2(512) = 9\) stages, each of which combines results pairwise via complex rotations. Plot the data flow and you get a pattern that looks like a butterfly — hence butterfly algorithm.

On the GPU this becomes a chain of fragment shader passes: each stage renders into a float framebuffer, the result is the input for the next shader. Ping-pong between two textures, nine passes, and at the end you get two output textures:

  • A displacement map — how far each point of the water surface is shifted up, down and sideways
  • A normal map — the surface normals for realistic light refraction

Bolting those two textures onto a Three.js water plane — that’s the entire integration job.

From paper to browser: the JavaScript implementation chain

By 2013 the maths was already nearly 30 years old. What was missing was a working implementation for the browser. That job was taken up by a series of open-source developers who didn’t invent anything new, but translated the Tessendorf method step by step into current web technologies:

  • 2013 — David Li builds a WebGL demo (david.li/waves/) that implements the Tessendorf math with fragment shaders instead of dedicated compute shaders
  • 2014 — Aleksandr Albert ports David Li’s code into the Three.js world
  • 2015 — Jérémy Bouny turns it into a reusable Three.js module (github.com/jbouny/ocean)
  • 2023 — Phil Crowther updates the module for current Three.js versions and publishes it at philcrowther.com/Aviation
  • 2023 — Attila Schroeder lifts the shaders to GLSL3 / WebGL2

None of these developers are FFT pioneers — they are all maintainers of a pipeline that brings Tessendorf’s 2001 paper, Mastin’s 1987 work and ultimately Phillips’s 1957 statistics into the browser. The module running in the simulator — Ocean3.js, about 570 lines — is the current iteration of that chain.

The integration with Claude Code

In classical work the integration into our simulator would probably have taken two days:

  • Understand Three.js framebuffer management
  • Document and map the constructor parameters
  • Recompute UVs of the water plane for tiling (our plane is 24 km wide, the wave tile only 2.4 km)
  • Trick Three.js’s Water class into accepting the FFT normal map instead of its default texture
  • Inject the FFT displacement map into the vertex shader via string replacement

With Claude Code it took under twenty minutes. The point isn’t that the AI understands the math more deeply than a textbook — but that it eliminates the mechanical work of integration. The human says: „Use this normal map instead of the default, and inject the displacement map into the vertex shader“ — and the AI finds the three places where that substitution needs to happen.

// WaterPlane.js — hybrid construction
const ocean = new Ocean(renderer, {
  Res: 512, Siz: 2400,
  WSp: 18, WHd: 295, Chp: 1.5
});

const water = new Water(geometry, {
  waterNormals: ocean.normalMapFramebuffer.texture, // FFT normal map
  sunColor: 0xffeedd,
  sunDirection: sunPos.normalize(),
});

// Inject displacement map into the vertex shader via shader injection
water.material.uniforms.oceanDisplacement = { value: ocean.displacementMapFramebuffer.texture };
water.material.vertexShader =
  'uniform sampler2D oceanDisplacement;\n' +
  water.material.vertexShader
    .replace('void main() {',
      'void main() { vec3 oceanDisp = texture2D(oceanDisplacement, uv).rgb;')
    .replace(/vec4\s*\(\s*position\s*,\s*1\.0\s*\)/g,
      'vec4(position + oceanDisp, 1.0)');

Eleven lines of glue code for an ocean simulation that computes 65,536 waves per frame in the background. The Water class provides the sun reflection, the FFT produces the physically grounded surface.

What was a trembling sheet of concrete is now a breathing ocean. During a low pass you can see individual wave crests catching and refracting the sunlight — each wave its own tiny mirror.

The transferable part

If you read this and have your own Three.js project, take one thing away: AI-assisted development has drastically lowered the barrier to adapting foreign graphics code. Understanding 570 lines of WebGL2 code with an FFT butterfly algorithm, float render targets and ping-pong buffers — and dropping it into someone else’s project — used to be a weekend project. Today it’s an evening.

The recipe is always the same: read the foreign code, understand the public surface (constructor, update method, output textures), then ask Claude Code to write the glue. The AI compensates for exactly what eats the most time in integrations like this: mechanical detail work, renaming variables, adapting signatures, finding the right shader lines.

Chapter 8

What This Says About AI and Software Development

Human and Machine

Let’s be honest: this project would have taken weeks without AI. Not because the concepts are hard — Three.js documentation is excellent, MediaPipe has examples. But because the iteration was so much faster.

The roles were clear:

The human provided vision, aesthetic judgment and the irreplaceable “this feels wrong.” No prompt in the world could have automated the realization that the wing flap felt “mushy.” Or that the trees looked like green sticks from above.

The AI provided architecture, boilerplate, debugging and an iteration speed no single developer can match. When the human said “more trees,” the AI added 100,000 — complete with octree optimization, LOD system and frustum culling so performance wouldn’t tank.

But — and this is the point — the AI didn’t improve the trees on its own. It placed 100,000 identical green cones and was satisfied. Only when the human said “those look like plastic” did texture variations, different tree types and color gradations appear.

The Future

In five years, any teenager will be able to build something like this in an afternoon. The tools will get better, the models smarter, the barrier to entry lower. But the decisive factor will remain the same: someone has to know what looks good, feels right and is fun.

The bird flies. But it doesn’t fly because the AI wanted it to on its own. It flies because a human said: “This needs to feel like flying.” And then iterated until it did.

Frequently Asked Questions

Can you really build a game with AI in one afternoon?

Yes – but with caveats. Claude Code wrote all the code, managed 40 tickets, and ran automated tests. The human was the Product Owner: vision, decisions, quality control. One afternoon is enough for an impressive prototype, not for a AAA game.

What is frustum culling?

Frustum culling is an optimization technique in 3D graphics. The camera sees only a cone-shaped slice of the world (the frustum). Everything outside does not need to be rendered. An octree helps quickly decide which objects are visible.

How does the webcam control work with MediaPipe?

MediaPipe detects 33 body landmarks in real-time via the webcam. Arm positions are converted to angles: raising and lowering creates wing flaps, lateral tilting controls direction, leaning forward triggers a dive.
